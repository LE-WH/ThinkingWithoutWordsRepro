## 4. Teacher traces 退化问题

**所有 5 个 trace 文件（pi1_phaseB 到 pi3_phaseB），全部 5000 条，trace 长度固定为 13 tokens。**

```
pi1_phaseB: min=13  median=13  max=13
pi2_phaseA: min=13  median=13  max=13
pi3_phaseA: min=13  median=13  max=13
（全部如此）
```

原因：`AbstractConstrainedLogits` 从第 1 个 token 起就把 `END_ABS` 放在 allowed 集合里，不强制最小长度。`m_min=16` 只用于 round 1 的**随机**初始 trace（`random_abstract_trace()`），on-policy 解码时没有下限约束。

模型收敛到固定 13 token 的 abstract trace，是 LoRA（17% 可训练参数）+ 5k 数据量不足以表达多样 Z̃ 的结果。Z̃ 没有携带有内容差异的信号，退化为一种固定 pattern。

最终 Z segment 长度（含 `<beginabstract>` / `<endabstract>` delimiter）：**固定 15 tokens**。

对比论文：论文 warm-up 后 mean total tokens 约 144（Z̃ + answer），而这次复现是 432。差距在 answer 更长 + Z̃ 压缩效率低。

## 5. 序列长度分析

### CoT 原始长度分布（N=500，从 HF 流式采样）

| 分位数 | CoT token 数 |
|---|---|
| p10 | 1,725 |
| p25 | 8,397 |
| **p50** | **20,016** |
| p90 | 28,295 |
| max | 32,707 |

CoT 中位数 **2 万 tokens**。Dolci-Think-SFT-7B 使用 DeepSeek-R1 风格生成，CoT 非常长。

### 在不同 max_len 下的截断情况

| max_len | 完整 fit | 被截断 |
|---|---|---|
| 2048（smoke run） | 9.8% | **90.2%** |
| 4096 | 16.4% | 83.6% |
| **8192（本次复现）** | **24.0%** | **76.0%** |

被截断的 380/500 条中，平均只保留 CoT 的前 **37%**（p50=34%）。
截断从右边切（`C = C[:budget_for_C]`），即丢掉的是 CoT 末尾——而结论、答案验证等关键步骤通常在末尾。

### Phase A / Phase B 实际序列长度（N=200）

| 段 | Phase A | Phase B |
|---|---|---|
| X（prompt 前缀）| p50=120 tok | p50=120 tok |
| C（verbal CoT）| p50=7,428 tok（截断后）| — |
| Z̃（abstract trace）| **固定 15 tok** | **固定 15 tok** |
| Y（answer）| p50=527 tok | p50=527 tok |
| **total** | **p50=8,192（76% 触顶）** | **p50=658 tok** |

Phase A 的 Z̃ 远小于 m_max=128，说明模型只学到了对截断后 CoT 前段的极简压缩。Phase B 序列很短（中位 658 tokens），计算很便宜。

## 7. 在 4× H200（140 GB each）上的完整复现可行性

### 硬件参数

| 指标 | A100-SXM4-80GB | H200-SXM-141GB |
|---|---|---|
| BF16 算力（dense） | 312 TFLOPS | ~495 TFLOPS（≈1.6×） |
| 显存带宽 | 2.0 TB/s | 4.8 TB/s（≈2.4×） |
| 显存 | 80 GB | 141 GB（≈1.76×） |

4× H200 vs 当前 2× A100：**综合训练加速约 3–3.2×**（计算主导时），**推理/vLLM 加速约 4–5×**（带宽主导）。

Qwen3-4B 有 32 个 attention head → TP 必须整除 32。4× H200 可以用 TP=4。

### 全参数微调的显存可行性（4× H200）

Qwen3-4B 全参数：4.86B
- bf16 模型权重：~10 GB
- Adam fp32 优化器状态（params + m + v）：~58 GB
- ZeRO-3 分片到 4 卡：每卡 (10+58)/4 ≈ **17 GB**
- seq=8192, micro_batch=4, gradient checkpointing 下激活：~8 GB/卡
- 每卡合计：约 **25–30 GB** → 140 GB 有非常充足的余量

**全参数微调在 4× H200 上完全可行，显存不是瓶颈。**

### 方案对比与时间估算

以下估算基于实测基准（Phase A：2× A100, LoRA, seq=8192, N=5k, 1epoch = 2.94 hr），线性外推并加 H200 加速系数。

#### 方案 A：LoRA + 60k samples + seq=16k + T=3 + 1 epoch
*数据量提升 12×，解决最严重的数据不足问题；seq=16k 让 ~45% CoT 完整 fit。*

| 阶段 | 每轮用时 | T=3 合计 |
|---|---|---|
| Phase A 训练 | ~3.5 hr | ~10.5 hr |
| Phase B 训练 | ~0.2 hr | ~0.6 hr |
| gen_traces Phase A（vLLM TP=4）| ~1.5 hr | ~3 hr（第 2/3 轮）|
| gen_traces Phase B | ~0.05 hr | negligible |
| Baseline eval + merge | — | ~0.5 hr |
| **总计** | | **~15 hr** |

预期结果：比当前 72% 有提升，但仍低于论文（数据量和 CoT 覆盖率均不及）。

---

#### 方案 B：Full FT + 600k samples + seq=8192 + T=3 + 1 epoch
*与论文数据量持平，但 seq 仍有截断（76% CoT 不完整），epoch 仍是 1。*

| 阶段 | 每轮用时 | T=3 合计 |
|---|---|---|
| Phase A 训练（全参数，大 batch） | ~22–28 hr | **~70–85 hr** |
| Phase B 训练 | ~2 hr | ~6 hr |
| gen_traces Phase A（600k, vLLM TP=4） | ~9–11 hr | ~20 hr（第 2/3 轮）|
| gen_traces Phase B | ~0.1 hr | ~0.3 hr |
| Baseline eval + merge | — | ~0.5 hr |
| **总计** | | **约 4–5 天** |

Phase A 的自定义 attention mask（全 2D mask）与 FlashAttention 不完全兼容，是主要性能瓶颈。gen_traces 的 Phase A 耗时因为 CoT prefix 长（~5500 tok），是第二瓶颈。

---

#### 方案 C：Full FT + 600k + seq=32k + T=3 + 1 epoch（最贴近论文）
*seq=32k 让 ~75% CoT 完整 fit（p75 以下的 CoT 全部保留）。*

注意：Phase A 的 attention 计算复杂度 ≈ O(seq²)。seq=32k vs seq=8k：理论上慢 16×，实践中 flash attention + 稀疏 mask 可以改善，但自定义 block mask 在极长序列下实现困难。

| 阶段 | 估算 |
|---|---|
| Phase A 训练（seq=32k, Full FT） | 每轮 **60–100 hr** |
| T=3 Phase A 合计 | **180–300 hr** |
| gen_traces Phase A（600k, 长前缀）| ~30 hr/round × 2 = 60 hr |
| **总计** | **~10–15 天** |

**结论：方案 C 在 4× H200 上时间不现实，除非把自定义 mask 换成 FlashAttention2 的 block-sparse 实现（需改动 `data_utils.py`）。**

---

### 推荐方案

| 目标 | 推荐方案 | 预估时间 |
|---|---|---|
| 快速验证（接近当前，更多数据） | 方案 A（LoRA + 60k + seq=16k） | **~15 hr** |
| 尽量接近论文（可接受数天）| 方案 B（Full FT + 600k + seq=8k） | **~4–5 天** |
| 完整论文复现 | 方案 C + FlashAttention block-sparse mask | **~10–15 天** |

**4× H200 上最现实的"接近论文"方案是方案 B（4–5 天）**，它解决了数据量不足（1/120 → 满量）和 LoRA 容量不足两个主要问题，代价是 CoT 截断仍存在（76% 被截断，但每条样本能看到前 34% CoT，比 2k seq 的 10% 已大幅改善）。

如果要进一步提升，需要修改 Phase A 的 attention mask 实现以支持长序列（FlashAttention 稀疏 mask），这是工程改动，不是简单参数调整。

### 9.2 `scripts/download_data.py` — 新增 `--max-total-tokens` 预过滤

**背景：** Dolci-Think-SFT-7B 的 CoT 中位数 20,016 tokens，在 max_len=8192 时 76% 样本被截断，平均只保留 CoT 前 37%，且截断从右边切（丢掉结论和答案验证步骤）。

**方案：** 下载时按估算 Phase A 序列长度过滤，只保留完整 fit 的样本，避免截断带来的质量损失。

**实测通过率（N=500 采样）：**

| max_len | 通过比例 | 600k 中约保留 |
|---|---|---|
| 8192 | 22% | 134k 条 |
| 16384 | **44%** | **266k 条** |
| 24576 | 90% | 538k 条 |

估算函数使用 `chars/3`（偏保守，实际 Qwen3 约 3.5 chars/tok），确保实际 token 数不超出限制。下载结束后自动打印 p10/p50/p90/max 分布供验证。

**用法：**
```bash
python scripts/download_data.py \
  --n 300000 --max-total-tokens 16384 \
  --dolci-out data/dolci_300k_16k.jsonl --skip-math500
```

`--max-total-tokens` 与训练的 `--max-len` 参数对应，传任意值均可（8192 / 16384 / 32768 等），无硬编码。

### 9.6 环境重建：torch 2.11 → torch 2.9.1

**背景：** flash-attn 2.8.3 预编译 wheel 最高只到 torch 2.9（无 torch 2.11 / cu13 wheel）；从源码编译因超时失败；cu12 wheel 因 `libcudart.so.12` ELF version symbol 不兼容无法在纯 CUDA 13 环境运行。

**解决方案：** 整体降级 vLLM（0.21.0 → 0.14.0），vLLM 的安装会将 torch 从 2.11 降到 2.9.1 并带入 `nvidia-cuda-runtime-cu12`（提供 `libcudart.so.12`），flash-attn cu12torch2.9 wheel 可正常安装。

**关键验证：**
- `allowed_token_ids` 在 vLLM 0.14.0 `SamplingParams` 中存在 ✓（dataclass field，`gen_traces.py` 可用）
- torch 2.9.1 vs 2.11.0 对训练结果无影响（BF16 matmul 在同一 GPU 上数值一致）

flash-attn 要求 SM80+（Ampere），因此在当前 V100 开发机上 flash-attn 已安装但 GPU forward 会报错。这是预期行为：代码开发和测试在 V100 上用 SDPA 进行，部署到 H200（SM90）时 flash-attn 自动生效。

### 9.7 Phase A FlashAttention 路径分析

**当前状态：** 训练使用 `attn_implementation="sdpa"`（PyTorch 原生），不用 FlashAttention。

**Phase B**（`[X; Z̃; Y]`，标准因果 mask）：改一行 `attn_implementation="flash_attention_2"` 即可，预计 2–3× 加速，但 Phase B 本来只需 6s/step，影响有限。

**Phase A**（`[X; C; Z̃; Y]`，自定义 2D block mask）：FA2 无法直接接受任意 dense mask。但 Phase A 的 mask 有等价的两 pass 分解：

```
Pass 1: [X; C; Z̃]  标准因果  → loss on Z̃   （Z̃ 学习压缩 C）
Pass 2: [X; Z̃; Y]  标准因果  → loss on Y    （Y 只看 Z̃，不看 C）
```

两者信息流等价于原始单 pass 自定义 mask。两 pass 均为标准因果，FA2 直接可用，预计 Phase A 加速 2–3×。需改动 `data_utils.py` + `train_phase_lora.py`，约 80–100 行，**尚未实现**。
